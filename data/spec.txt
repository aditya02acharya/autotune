# Guardrails Service — Requirements (v1)

> **Scope:** this document covers the **guardrails service only** — the component that
> applies guardrails to a request in active or passive mode. The gateway, policy routing,
> and the 10k-RPS ingress live elsewhere and are out of scope.
>
> **v1 principle: keep it simple.** No Redis, no cross-pod coordination, no durable queue.
> Everything is in-process. Capacity-sharing improvements are deferred to v2 (§7).

---

## 1. What this service does

Receives a request payload and applies guardrails, in one of two modes:

- **Active (prevention):** synchronous. Caller blocks and receives an allow / block /
  redact decision before proceeding. This is a gate.
- **Passive (detection / audit):** asynchronous. Service acks immediately; the guardrail
  runs out of band and the result is logged/emitted. It **cannot** block the caller's
  response (already sent), so it is inherently **best-effort** and may shed under load.

Both modes call the same rate-limited upstream (Bedrock Guardrails) and therefore draw
from the same finite budget.

---

## 2. Topology (fixed)

| Element | Value | Consequence |
|---|---|---|
| EKS pod replicas | 2 | Two failure domains |
| Uvicorn workers/pod | 4 | **8 independent OS processes** |
| Per process | own event loop, own httpx pool, own semaphores | In-process limits do **not** coordinate across workers |
| Shared store | **none in v1** | Global budget must be split statically (§4) |

---

## 3. The binding constraint — Bedrock Guardrails

`ApplyGuardrail` default quota (us-east-1 / us-west-2), both adjustable via Service Quotas:

- **50 calls/sec**
- **200 text units/sec** (1 TU ≈ 1,000 chars)

Which one binds: `min( 50, 200 / avg_TU_per_call )`. Payloads > ~4 KB hit the TU limit first.

Little's Law sanity check: `50/sec × ~0.3 s ≈ ~15` concurrent calls in flight. Small and
trivially bounded — there is no large-concurrency problem on the guardrail path itself.

**Capacity reality for v1:** the service can sustain ~48 guardrail calls/sec total (§4).
Any guardrail-requiring demand above that is **shed** (active) or **dropped** (passive).
v1 is correct and safe at this rate; raising real throughput needs a quota increase, a
local classifier offload, or both — see §7 and §8.

---

## 4. Global budget without a shared store — static partition

The only way to guarantee the aggregate never exceeds Bedrock's quota, with no shared
store, is to give each worker a fixed slice:

```
calls:       50 / 8 = 6.25  → 6 calls/sec per worker   → 48/sec aggregate (safe < 50)
text units: 200 / 8 = 25 TU/sec per worker
```

Enforced by a simple **in-process token bucket** per worker (one for calls, one for TUs).
No network hop, no dependency.

- **Tradeoff:** under uneven load some workers throttle while the global budget isn't
  fully used. Accepted for v1.
- **Why conservative rounding:** sum of per-worker limits must be ≤ 50, so round *down*.

---

## 5. v1 Requirements

**R1 — Budget enforcement (in-process).** Per-worker token buckets sized per §4. Retries
draw from the same bucket so they cannot bypass the limit.

**R2 — Per-upstream isolation.** Dedicated httpx connection pool, concurrency semaphore,
timeout budget, and circuit breaker for Bedrock (and for the local model if used). No
shared unconstrained pool.

**R3 — Strict timeouts.** Connect, pool-acquire, read, write, and an overall deadline on
every upstream call. A slow upstream must never hold a request open indefinitely.

**R4 — Bounded concurrency.** Explicit caps on in-flight active requests per worker,
upstream concurrency, and passive queue depth. On limit reached: shed/defer, never run
unbounded.

**R5 — Retry safety.** Bounded retries, exponential backoff **with jitter**,
idempotency-aware, budget-aware (R1). Active and passive retry policies configured
separately; on Bedrock 429, back off rather than hammer.

**R6 — Graceful degradation (fail *closed* for a safety service).**
- Active + budget exhausted / breaker open → **reject** (e.g. 429/503). Do **not** allow
  unchecked — failing open defeats a guardrails service.
- Passive + queue full → drop with a metric (acceptable by design; passive is best-effort).
- A per-class override may permit fail-open only for explicitly low-risk traffic.

**R7 — Passive queue (in-memory, bounded).** A bounded `asyncio` queue per worker drains
at the worker's budget rate. Smooths short bursts only; sheds when full. **Not durable** —
in-flight passive work is lost on restart (acceptable for best-effort detection in v1).

**R8 — Observability.** Emit: budget utilisation (calls + TU), throttle/shed counts,
upstream latency percentiles, 429 rate, breaker state, passive queue depth, active
reject rate. These metrics are what justify the v2 work.

---

## 6. Components (v1)

| Component | Responsibility |
|---|---|
| HTTP handler | Accept request, branch active vs passive |
| Budget limiter | Per-worker token buckets (calls + TU) |
| Bedrock client | Bounded-concurrency call + timeouts + breaker |
| Passive runner | In-memory bounded queue + background drain at budget rate |
| (Local classifier) | *Optional / TBD — see §8* |
| Metrics | Counters + histograms per R8 |

Keep budget + breaker logic in one small, testable module rather than scattered through
handlers.

---

## 7. Explicitly deferred to v2

- **Redis token bucket** — one shared budget across all 8 workers; recovers the capacity
  lost to static partitioning (§4) and lets the service actually use the full 50/sec.
- **Durable passive queue** (e.g. Redis Streams) — survives restarts, retry isolation,
  independent worker scaling, lag metrics.
- **Cross-pod / multi-region coordination** — only if quota is raised substantially or
  the deployment spans regions.

---

## 8. Open items (don't block v1)

1. **Role of the 1B vLLM model** — local guardrail classifier (fast-path that resolves
   most requests and escalates only uncertain ones to Bedrock) vs. the model being
   protected. If it's a classifier, it's the main lever for closing the 10k → budget gap
   and would slot in as the "Local classifier" component in §6.
2. **Sustained guardrail-requiring rate** — sets whether ~48/sec is adequate for v1 or a
   quota increase is needed before launch.
3. **Average payload size** — decides whether the 50/sec or 200 TU/sec limit binds, i.e.
   which token bucket throttles first.
